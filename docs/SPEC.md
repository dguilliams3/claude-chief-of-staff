# Chief of Staff — System Specification

> Canonical reference for API routes, data model, and sync protocol.
> For project conventions and task execution, see `CLAUDE.md`.

---

## System Overview

A briefing pipeline with three layers:

1. **Agent runner** (local) — invokes `claude --print` to generate briefings from MCP data sources
2. **Local API** (Hono on Node, port 3141) — serves the agent runner, exposed via cloudflared tunnel
3. **Cloud API** (Hono on Cloudflare Workers) — D1-backed CRUD + tunnel proxy, consumed by the PWA

Data flows one direction: agent → local file → sync POST → D1. The PWA reads from D1 (briefings, history) and proxies through the Worker → tunnel → local API for live operations (trigger, follow-up).

## API Routes

All authenticated routes require `Authorization: Bearer <COS_TOKEN>`.

### Cloud API (Cloudflare Worker)

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/health` | No | Health check (setup wizard validation) |
| GET | `/briefings/types` | Yes | Static briefing type metadata (labels, descriptions) |
| GET | `/briefings/latest` | Yes | Latest briefing per type (keyed by type) |
| GET | `/briefings/` | Yes | History list — 50 most recent, summary fields only |
| GET | `/briefings/:id` | Yes | Single briefing with full sections and metadata |
| POST | `/briefings/sync` | Yes | Upsert briefing to D1 (INSERT OR REPLACE) |
| POST | `/briefings/trigger` | Yes | Proxy → tunnel → local API → starts briefing generation |
| GET | `/briefings/status/:jobId` | Yes | Proxy → tunnel → local API → poll trigger job status |
| POST | `/briefings/follow-up` | Yes | Persist user message → proxy → tunnel → persist assistant response |
| GET | `/conversations` | Yes | List conversations with message counts (Chats tab) |
| GET | `/conversations/:id/messages` | Yes | Messages for a conversation (limit 200) |
| GET | `/conversations/by-briefing/:briefingId` | Yes | Look up conversation by associated briefing |

### Local API (Hono on Node, via tunnel)

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/briefings/trigger` | Yes | Starts a briefing generation job (async, returns jobId) |
| GET | `/briefings/status/:jobId` | Yes | Poll job completion status |
| POST | `/briefings/follow-up` | Yes | Invokes `claude --print --resume` for follow-up Q&A |

### Response Conventions

All responses use camelCase field names regardless of D1 column naming (snake_case in SQL, mapped at the route handler). Timestamps are ISO 8601 strings. Briefing `sections` and `metadata` are stored as JSON TEXT columns, parsed and returned as objects.

## Data Model

### D1 Tables

**briefings**
| Column | Type | Notes |
|--------|------|-------|
| id | TEXT PK | UUID, generated by agent runner |
| type | TEXT NOT NULL | Briefing type key (e.g., `work`, `news`) |
| generated_at | TEXT NOT NULL | ISO 8601 timestamp |
| session_id | TEXT NOT NULL | Claude CLI session ID (for `--resume`) |
| sections_json | TEXT NOT NULL | Stringified JSON array of section objects |
| metadata_json | TEXT NOT NULL | Stringified JSON object (briefingNumber, etc.) |
| created_at | TEXT | D1 default `datetime('now')` |

Index: `idx_briefings_type_generated(type, generated_at DESC)`

**sessions**
| Column | Type | Notes |
|--------|------|-------|
| id | TEXT PK | Claude CLI session ID string |
| created_at | TEXT NOT NULL | Default `datetime('now')` |

**conversations**
| Column | Type | Notes |
|--------|------|-------|
| id | TEXT PK | UUID |
| session_id | TEXT NOT NULL UNIQUE | FK → sessions.id |
| briefing_id | TEXT UNIQUE | FK → briefings.id (nullable) |
| created_at | TEXT NOT NULL | Default `datetime('now')` |

**messages**
| Column | Type | Notes |
|--------|------|-------|
| id | TEXT PK | UUID |
| conversation_id | TEXT NOT NULL | FK → conversations.id |
| role | TEXT NOT NULL | CHECK constraint: `user` or `assistant` |
| content | TEXT NOT NULL | Message text |
| created_at | TEXT NOT NULL | Default `datetime('now')` |

Index: `idx_messages_conversation_created(conversation_id, created_at)`

### Append-Only Invariant

These tables never receive UPDATE or DELETE operations (except briefings, which uses INSERT OR REPLACE for idempotent sync). If PRAGMA foreign_keys is ever enabled, the INSERT OR REPLACE on briefings will fail (it does DELETE+INSERT, triggering FK violations on conversations.briefing_id).

## Sync Protocol

After the agent runner generates a briefing locally:

1. `run-briefing.ts` writes the briefing JSON to a local file (transient, not durable)
2. `sync.ts` POSTs the briefing object to `${COS_WORKER_URL}/briefings/sync`
3. The Worker route does `INSERT OR REPLACE` into D1
4. After persisting the briefing, `ensureConversation()` creates session + conversation rows if they don't exist (non-fatal on failure)

Sync is **non-fatal**: if `COS_WORKER_URL` or `COS_TOKEN` is unset, or the POST fails, the briefing still exists locally and can be synced later. The system logs a warning but does not abort.

## Follow-Up Flow

When a user asks a follow-up question via the PWA:

1. PWA sends `POST /briefings/follow-up` to the Worker with `{ sessionId, question, briefingId? }`
2. Worker persists the user message to D1 (synchronous, non-fatal on failure)
3. Worker proxies to tunnel → local API
4. Local API invokes `claude --print --resume <sessionId>` with the question
5. Worker persists the assistant response to D1 (synchronous, non-fatal on failure)
6. Worker returns `{ answer, sessionId, userMessage, assistantMessage }` to PWA

The assistant message UUID and timestamp are generated at the Worker before the tunnel call, ensuring the same ID is used in both the D1 write and the PWA response.

## Briefing Type System

Briefing types are registered in `agent/registry.ts`. Each type provides a prompt config (composed from reusable prompt components), a display label, a timeout, and a list of data sources sampled.

Adding a new type requires two edits:

1. Create `agent/briefings/<type>/config.ts` — export a `Prompt` config
2. Edit `agent/registry.ts` — import and register the config

The API returns available types at `GET /briefings/types` from a shared JSON file (`shared/briefing-types.json`). The PWA fetches this on init to know what tabs to render.

## Authentication

Single bearer token (`COS_TOKEN`) shared between all layers:

- **Worker**: stored as a Cloudflare secret (`wrangler secret put COS_TOKEN`), checked by `middleware/auth.ts`
- **Local API**: loaded from `.env`, checked by inline middleware in `server.ts`
- **PWA**: user enters token on first load, stored in Zustand (persisted to memory)

The `/health` endpoint is the only unauthenticated route (used by setup wizard validation).

Token rotation: generate new token → update Worker secret → update `.env` → re-enter on PWA.
